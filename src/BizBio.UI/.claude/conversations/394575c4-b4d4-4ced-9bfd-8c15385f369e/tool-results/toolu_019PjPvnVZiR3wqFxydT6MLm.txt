    15→## Architecture Overview
    16→
    17→### The Two-Layer Approach
    18→
    19→```
    20→┌─────────────────────┐
    21→│   Frontend/API      │  ← MRD-style DTOs (efficient, indexed)
    22→├─────────────────────┤
    23→│  Mapping Service    │  ← Converts between layers
    24→├─────────────────────┤
    25→│  Database/EF Core   │  ← Normalized relational schema
    26→└─────────────────────┘
    27→```
    28→
    29→**Why this works:**
    30→- ✅ Database stays normalized (easy to query, update, maintain)
    31→- ✅ API is efficient (single response, minimal data, fast parsing)
    32→- ✅ Frontend is simple (no complex lookups, easy cart logic)

    36→## Database Schema
    37→
    38→### Your Current Entities (Normalized)
    39→
    40→```csharp
    41→CatalogItem
    42→├── Variants[]
    43→│   ├── Id, Title, Price
    44→│   └── IsDefault
    45→└── ExtraGroupLinks[]
    46→    ├── ExtraGroup
    47→    │   ├── Name, MinRequired, MaxAllowed
    48→    │   └── GroupItems[]
    49→    │       └── Extra (Id, Name, BasePrice)
    50→    └── VariantId (nullable - applies to specific variant or all)
    51→```
    52→
    53→### Key Relationships
    54→
    55→```sql
    56→-- Items have variants (size/price options)
    57→CatalogItemVariant
    58→  - CatalogItemId → CatalogItem
    59→  - Price, Title, IsDefault
    60→
    61→-- Items are linked to extra groups
    62→CatalogItemExtraGroupLink
    63→  - CatalogItemId → CatalogItem
    64→  - ExtraGroupId → CatalogItemExtraGroup
    65→  - VariantId (optional - variant-specific extras)
    66→
    67→-- Extra groups contain individual extras
    68→CatalogItemExtraGroupItem
    69→  - ExtraGroupId → CatalogItemExtraGroup
    70→  - ExtraId → CatalogItemExtra
    71→```

    75→## The MRD Pattern
    76→
    77→### What Makes It Special
    78→
    79→Instead of this (traditional):
    80→```json
    81→{
    82→  "variant": {
    83→    "id": 123,
    84→    "options": [
    85→      { "id": 1, "name": "Coke", "price": 0 },
    86→      { "id": 2, "name": "Sprite", "price": 0 }
    87→    ],
    88→    "extras": [
    89→      { "id": 10, "name": "Extra Cheese", "price": 5 }
    90→    ]
    91→  }
    92→}
    93→```
    94→
    95→MRD does this (indexed):
    96→```json
    97→{
    98→  "options": [
    99→    { "id": 1, "name": "Drinks", "items": [...] },
   100→    { "id": 2, "name": "Condiments", "items": [...] }
   101→  ],
   102→  "extras": [
   103→    { "id": 10, "name": "Upsells", "items": [...] }
   104→  ],
   105→  "variants": [
   106→    {
   107→      "id": 123,
   108→      "option_indices": [0, 1],  // Uses options[0] and options[1]
   109→      "extra_indices": [0]        // Uses extras[0]
   110→    }
   111→  ]
   112→}
   113→```
   114→
   115→### Benefits
   116→
   117→1. **No Data Duplication**
   118→   - Options/extras defined once
   119→   - Referenced by index across all variants
   120→   - Smaller payload size
   121→
   122→2. **Faster Frontend**
   123→   - No nested lookups
   124→   - Direct array access: `menu.options[index]`
   125→   - Simple cart logic
   126→
   127→3. **Better Caching**
   128→   - Entire menu structure stable
   129→   - Only prices/availability change
   130→   - Easy to cache and invalidate
   131→

   134→## API Implementation
   135→
   136→### Step 1: Menu Response Structure
   137→
   138→```csharp
   139→public class MenuResponseDto
   140→{
   141→    public List<MenuSectionDto> Sections { get; set; }
   142→    public List<ExtraGroupDto> Extras { get; set; }    // ← Global array
   143→    public List<OptionGroupDto> Options { get; set; }  // ← Global array
   144→}
   145→
   146→public class VariantDto
   147→{
   148→    public int Id { get; set; }
   149→    public decimal Price { get; set; }
   150→    public List<int> OptionIndices { get; set; }  // ← References Options[]
   151→    public List<int> ExtraIndices { get; set; }   // ← References Extras[]
   152→}
   153→```
   154→
   155→### Step 2: Mapping Service
   156→
   157→The `MenuMappingService` converts your normalized DB to MRD format:
   158→
   159→```csharp
   160→// 1. Build global options/extras arrays
   161→var allExtraGroups = catalog.Items
   162→    .SelectMany(item => item.ExtraGroupLinks)
   163→    .Select(link => link.ExtraGroup)
   164→    .DistinctBy(g => g.Id);
   165→
   166→// 2. Separate into Options vs Extras
   167→//    Options: MinRequired >= 1 (e.g., "Choose a drink")
   168→//    Extras: MinRequired = 0 (e.g., "Add extra cheese?")
   169→foreach (var group in allExtraGroups)
   170→{
   171→    if (group.MinRequired >= 1)
   172→        optionGroups.Add(group);  // Index 0, 1, 2...
   173→    else
   174→        extraGroups.Add(group);   // Index 0, 1, 2...
   175→}
   176→
   177→// 3. Build variant with indices
   178→variant.OptionIndices = item.ExtraGroupLinks
   179→    .Where(link => link.ExtraGroup.MinRequired >= 1)
   180→    .Select(link => optionIndexMap[link.ExtraGroupId])
   181→    .ToList();
   182→```
   183→
   184→### Step 3: API Endpoint
   185→
   186→```csharp
   187→[HttpGet("api/v2/menu/{catalogId}")]
   188→public async Task<ActionResult<MenuResponseDto>> GetMenu(int catalogId)
   189→{
   190→    var catalog = await LoadCatalogWithIncludes(catalogId);
   191→    var menuDto = _menuMapper.MapCatalogToMenu(catalog);
   192→    return Ok(menuDto);
   193→}
   194→```

   198→## Frontend Integration
   199→
   200→### Example: Vue/Nuxt Cart Store
   201→
   202→```typescript
   203→// stores/cart.ts
   204→interface CartItem {
   205→  variantId: number
   206→  quantity: number
   207→  selectedOptions: number[]  // Array of option IDs
   208→  selectedExtras: Array<{ id: number, quantity: number }>
   209→}
   210→
   211→// When user adds item to cart
   212→function addToCart(item: MenuItem, variant: Variant) {
   213→  const selectedOptions: number[] = []
   214→
   215→  // For each option group this variant uses
   216→  variant.option_indices.forEach(index => {
   217→    const optionGroup = menu.options[index]  // ← Direct array access!
   218→
   219→    // Show UI for user to select from optionGroup.items
   220→    const selectedOption = await showOptionPicker(optionGroup)
   221→    selectedOptions.push(selectedOption.id)
   222→  })
   223→
   224→  // Same for extras
   225→  variant.extra_indices.forEach(index => {
   226→    const extraGroup = menu.extras[index]
   227→    // Show optional extras picker
   228→  })
   229→
   230→  cart.items.push({
   231→    variantId: variant.id,
   232→    quantity: 1,
   233→    selectedOptions,
   234→    selectedExtras
   235→  })
   236→}
   237→```
   238→
   239→### Example: Calculating Price
   240→
   241→```typescript
   242→function calculateItemTotal(cartItem: CartItem): number {
   243→  // 1. Get base variant price
   244→  const variant = findVariantById(cartItem.variantId)
   245→  let total = variant.price
   246→
   247→  // 2. Add option costs (if any)
   248→  cartItem.selectedOptions.forEach(optionId => {
   249→    const option = findOptionById(optionId)  // Simple ID lookup
   250→    total += option.price
   251→  })
   252→
   253→  // 3. Add extras costs
   254→  cartItem.selectedExtras.forEach(({ id, quantity }) => {
   255→    const extra = findExtraById(id)
   256→    total += extra.price * quantity
   257→  })
   258→
   259→  return total * cartItem.quantity
   260→}
   261→```

   265→## Cart Logic
   266→
   267→### Validating Cart on Backend
   268→
   269→```csharp
   270→[HttpPost("api/v2/menu/calculate-cart")]
   271→public async Task<ActionResult> CalculateCart(CartCalculationRequest request)
   272→{
   273→    decimal total = 0;
   274→
   275→    foreach (var item in request.Items)
   276→    {
   277→        // 1. Validate variant exists
   278→        var variant = await _context.CatalogItemVariants
   279→            .Include(v => v.CatalogItem)
   280→                .ThenInclude(i => i.ExtraGroupLinks)
   281→            .FirstOrDefaultAsync(v => v.Id == item.VariantId);
   282→
   283→        if (variant == null)
   284→            return BadRequest("Invalid variant");
   285→
   286→        // 2. Validate selected options
   287→        foreach (var optionId in item.SelectedOptions)
   288→        {
   289→            var option = await _context.CatalogItemExtras
   290→                .FindAsync(optionId);
   291→            if (option == null)
   292→                return BadRequest($"Invalid option: {optionId}");
   293→            total += option.BasePrice;
   294→        }
   295→
   296→        // 3. Validate extras
   297→        foreach (var extra in item.SelectedExtras)
   298→        {
   299→            var extraItem = await _context.CatalogItemExtras
   300→                .FindAsync(extra.ExtraId);
   301→            if (extraItem == null)
   302→                return BadRequest($"Invalid extra: {extra.ExtraId}");
   303→            total += extraItem.BasePrice * extra.Quantity;
   304→        }
   305→
   306→        total += variant.Price;
   307→    }
   308→
   309→    return Ok(new { subtotal = total });
   310→}
   311→```

   417→## Key Advantages Over Traditional Approach
   418→
   419→### Traditional (Nested)
   420→- ❌ Each variant duplicates options/extras data
   421→- ❌ Larger payload (10 variants × 5 options = 50 duplicates)
   422→- ❌ Frontend needs nested lookups
   423→- ❌ Harder to cache
   424→
   425→### MRD Pattern (Indexed)
   426→- ✅ Options/extras defined once
   427→- ✅ Variants reference by index (tiny arrays: `[0, 1, 2]`)
   428→- ✅ Smaller payload (~50% reduction)
   429→- ✅ Direct array access: `menu.options[0]`
   430→- ✅ Easy to cache

   434→## Database Migration Notes
   435→
   436→Your existing schema already supports this! You have:
   437→- ✅ `CatalogItemExtraGroup` (option/extra groups)
   438→- ✅ `CatalogItemExtra` (individual options/extras)
   439→- ✅ `CatalogItemExtraGroupLink` (links items to groups)
   440→- ✅ `VariantId` nullable field (variant-specific groups)
   441→
   442→**No database changes needed!** Just:
   443→1. Use `MinRequired >= 1` to identify "options" (required)
   444→2. Use `MinRequired = 0` to identify "extras" (optional)
   445→3. Map to MRD-style DTOs in the service layer

   451→You get the best of both worlds:
